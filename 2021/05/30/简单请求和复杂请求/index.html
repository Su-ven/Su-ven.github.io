简单请求和复杂请求

简单请求和复杂请求

说到这个，需要从跨域谈起

同源策略

这个是一个默认的策略，也是浏览器最早引入的时候，为了保证基本的信息安全而提出的一个策略，其表现为"协议， 域名， 端口"三者皆相同，即使不同的域名最终指向同一个IP，但是在表现上，它们依然会被认为是非同源。浏览器请求，缓存, DOM操作等都需要遵循同源策略。

跨域

在当前源下，指向了另外一个不同的源，即产生了跨域行为。跨域只跟浏览器相关，浏览器默认的一种安全策略下的行为模式。如果使用服务端发送跨域http请求，并不会产生所谓的跨域问题

跨域解决方案:

  • CORS
  • JSONP
  • Nginx转发
  • 代理服务器转发
  • document.domain + iframe
  • location.hash + iframe
  • window.name + iframe
  • webSocket
  • postMessage

CORS

mdn
CROS(Cross-origin resource sharing)全称“跨域资源共享”，是为一种W3C标准，是一种基于HTTP头的机制。
它允许浏览器向不同源的服务端发送请求，以获取相应的数据或资源。
如果返回响应头中不设置Access-Control-Allow-Origin相应值的话，当前请求会被浏览器以跨跨域警报给拦截
CORS机制把请求区分为简单请求和复杂请求两种情况

对于有可能会造成服务端风险的HTTP请求方法，浏览器必须先发送一个OPTIONS方法，发起一次预请求，服务端返回通过之后，才会再继续发起实际HTTP请求。通过预请求返回，服务端也可以告知客户端是否需要携带标识信息(Cookie之类)

  • 简单请求:
    请求类型仅限: POST, GET, HEAD三种

    只支持下面的请求头关键字:
    Accept, Accept-language, content-language, content-type, DPR, Downlink, Save-Data, Viewport-Width, Width

    其中content-type的值仅限三种: text/plain, multipart/form-data, application/x-www-from-urlencode

  • 复杂请求
    去除简单请求则是复杂请求，表现为通过chrome查看的时候，可以发现一条请求会发送两次，一个类型为OPTIONS, 第二次才是实际请求

XMLHttpRequest 是遵循同源策略的，而一旦产生跨域行为，则会遵循 CORS 机制， 这是浏览器的约定默认行为。当跨域产生的时候，因为非同域，很容易会产生被攻击的风险。所以私以为，复杂请求的预请求的产生，主要目的是为了降低风险。当请求被劫持擅改后，头信息会被添加额外信息，此时就需要服务端确认当前请求是否合理安全，通过服务器预检后，才会发送正常的数据请求，否则该请求会被拦截掉(401等)

JSONP

前文说的CORS技术是可以达到支持跨域的目的的了，而且还无需做任务配置，仅需服务端提供支持即可。然而，在老旧版本的浏览器中，并没有提供对CORS的支持，在最初始的时候，跨域使用的是JSONP技术。

  • JSONP原理
    根据HTML的原理，在 script 标签中加载一个不同域的JS资源请求，也是可以正常下载并执行的。因此，利用这个规则，把可执行函数以params的方式返回前端，包装成script src的形式，即可实现跨域执行。

大体逻辑即是: 先定制好像全局执行函数callback，通过动态创建script的方式，把请求发往后端，与后端约定接收callback函数的key，后端处理完相关逻辑后，以执行函数的方式，返回前端。前端在loading资源后，会自动执行脚本代码，触发全局函数callback,以此达到跨域的目的。

缺点: 由于是通过创建script的方式，请求只能以GET的方式进行。
优点: 因为是利用的浏览器特性，可以在古老不支持CORS的浏览器上做跨域方案

Nginx转发 \ 代理服务器转发

两者的原理都一样。前文有说过，同源策略是为浏览器的安全策略，而非是HTTP协议。那么只要不在浏览器中发起不同域的请求，而由服务器或者代理服务器进行中间转发，就跨过了跨域问题。

document.domain + iframe

浏览器策略中，非同源的情况可引用资源，但是不可操作脚本。而当双方基础域名相同的话，就可以相互操作。 domain可在被一定范围内被设置。domain可被设置为当前域名或者是基础域名。利用两个特性结合，就可以实现了domain+iframe来达成跨域资源通信。

但是私以为，这种方案在目前的技术实现下其实并没有什么卵用，通常如果是属于后端的数据处理逻辑不会放到前端来，而跨页面通讯可以通过localhost或者spa组件化等解决方案替代。iframe在某些浏览器下有会被屏蔽的风险

location.hash + iframe

实现方式和上面的差不太多，由于同源策略，可以一个中间页面做为中间层，a的iframe是b，b的iframe是c，a页面把请求发往b页面，b页面把请求透传到c页面，由于c页面与a页面同源，因此可以运用parent.parent去调a页面的方法

也是骚操作

postMessage

postMessage是HTML5 XMLHttpRequest Level 2中的新增API，它可以安全的实现跨源通信。
