vue 编译原理

注：如果是通过 render 函数进行编写的话，那是不会走编译的方法的。

说到这里，需要首先了解下VUE的版本区别，其实在官方文档上也有详细的说明
大体上构建版本可以分为下面几种：

UMD CommonJS ES Module (基于构建工具使用) ES Module (直接用于浏览器)
完整版 vue.js vue.common.js vue.esm.js vue.esm.browser.js
只包含运行时版 vue.runtime.js vue.runtime.common.js vue.runtime.esm.js -
完整版 (生产环境) vue.min.js - - vue.esm.browser.min.js
只包含运行时版 (生产环境) vue.runtime.min.js - - -

详细可参考下官网 Vue

以vue-cli为例，使用inspect弹出配置项后，可以查看到以下配置

可见脚手架中使用的是vue的运行时版本，然而我们在直接引用运行时版本，写入一个template的时候，会被抛出一个错误提示

vue.runtime.js:625 [Vue warn]: You are using the runtime-only build of Vue where the template compiler is not available. Either pre-compile the templates into render functions, or use the compiler-included build.

(found in \<Root>)

大致意思是当前VUE为运行时版本，编译器不可用。

既然分了两种情况，那我们就分两步走。先看在vue-cli脚后架下的vue编译情况

vue-loader

\node_modules\vue-loader\lib\index.js

vue-loader 是用于对vue后缀的文件进行解析处理的
具体原理先略过，可以看到源码中有引用了vue-template-compiler

我们把眼光转到vue-template-compiler中去。

vue-template-compiler

node_modules\vue-template-compiler\build.js

上面代码可以知道，vue-template-compiler 版本一定需要和vue版本保持一致

可以看到baseCompile方法的处理流程

  • 先通过parse方法把模板转化为ast树
  • 再通过optimize方法进行优化处理(无需要动态的节点将会打上静态标签)
  • generate 用于生成对应的可执行代码

parseHTML

parse 方法对应的处理html的核心即是parseHTML方法
基本逻辑是根据正则匹配的方式，匹配出模板中的各个标签类，再通过对应的勾子进行操作
最后返回一个ast树
ast树基本结构:

其中通过 chars 方法，对每个节点的内容进行对应的处理。通过正则表达式 /\{\{((?:.|\r?\n)+?)\}\}/g 筛选出变量类的内容，并打标记为 type: 2，正常文本则打标记为 type: 3

optimize

optimize 方法主要做了两个动作，一是标记所有非静态节点，还有一个是标记所有的静态节点

先看markStatic

isStatic 方法是为做基础的静态节点判断。如为表达式(type === 2)，返回false表明非静态节点，如为文本(type === 3)，返回true表明为静态节点。如两都不是，则说明当前为一个标签，可通过相应的属性等进行判断是否为静态节点。
初步判断完后，根据节点是否含有children，进行递归处理。如果当前子节点为动态节点，则父节点也是为动态节点

generate

generate 方法里有一个比较重要的方法genElement, 这个方法主要作用，就是根据传入的ast树，以及各种状态，将ast树转化为对应的可执行的render字符串

然后会在高级函数createCompileToFunctionFn中，通过createFunction方法，以new Function的方式, 将可执行代码字符串转化为可执行方法

脚手架与vue编译版本

两者间的转化逻辑并没有什么不同。翻代码就可以知道，vue的编译版本中走的也是 vue-template-compiler 一套，可以认为vue-template-compiler是由vue源码中抽离出去的一段特定逻辑代码。

但是通过脚手架编辑，是为预编译，在vue运行过程中无需再重新进行模板的编译操作，相对来说其性能会更高一些。
