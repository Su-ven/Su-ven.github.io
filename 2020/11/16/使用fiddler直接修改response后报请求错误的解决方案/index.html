使用fiddler直接修改response后报请求错误的解决方案

发表于 2020-11-16 | 分类于 前端
fiddler修改返回

场景

使用fiddler工具直接修改接口返回，不使用文件的方式(按F2直接修改response)，偶尔会出现这种场景：执行save后，接口直接没有返回了，浏览器报Network Error。

原因

这时候查看请求头，可以发现有这么一个属性transfer-encoding: chunked

查看fiddler中抓回的response，发现多了多余字符

这个就是引起了错误请求的原因

解决

F2后，在弹出的编辑窗口选择header页签，然后找到transfer-encoding: chunked这个属性，右键，remove header。然后选择textView页签，把产生的多余编码符删除，就可以正常进行response编辑了

关于transfer-encoding属性

在通过HTTP传送数据的时候，有时并不能事先确定body的长度，因而无法直接得到Content-length的长度，所以无法在header中指定Content-length。那么要怎么判断发送方已经完成了报文报送呢？HTTP/1.1协议在header中引入了transfer-encoding，当其值为chunked的时候，表明其采用了chunked的编码方式进行报文传输。chunked编码的基本方式是将大块数据分解成多块小数据，每块都可以自定义长度。

HTTP/1.1 默认采用的是持久连接(chrome可以在 Response Headers中使用 view source查看版本，也可以通过插件或者fiddler之类的工具获取)，就是说一次TCP之后不会马上释放，并且允许多个请求响应在一个TCP连接上进行发送，所以客户端与服务端需要有一个标识来表明报文的结束点以及上一个报文的起始点。最简单的方法是使用Content-length，但是如果报文长度无法进行预先判断，对于动态的内容或者发送数据之前无法判定长度，可以使用分块的方式进行编码发送
